# 06 — Open Questions (Decision Gates for the Next Iteration)

**Purpose.** The questions that must be answered before the next extension spike or production build starts. Arranged so a reviewer can walk top-to-bottom with the user, capturing answers as a short addendum. Do not edit this file with the answers — this archive is append-only; add the answers as a sibling `06-open-questions-answers.md` when ready.

Each question links back to the report that justifies it.

---

## Q1. Who authors extensions? (ref: [`01-feasibility.md`](01-feasibility.md) §1, [`03-unbiased-approach.md`](03-unbiased-approach.md) §1)

Pick the set of intents in scope for v1:

- [ ] **Only us** — we ship a curated library; users toggle.
- [ ] **Power users pasting snippets** — users install text-form extensions from trusted sources (us, agent-stuff repo, blog posts).
- [ ] **Any user publishing** — third-party marketplace (out of scope, but flagging in case this is a hidden expectation).

Default if unanswered: first option only. This is by far the most consequential choice; everything downstream bends to it.

---

## Q2. Is DOM-rendering an extension genre we want? (ref: `01-feasibility.md` §2, Axis I.D)

Custom message renderers, chart embedders, Mermaid, etc. require iframe sandboxing for safety. They are a different product from handler-style extensions.

- [ ] **Yes, in v1** — plan for an iframe sandbox path.
- [ ] **Yes, later** — defer; the v1 runtime should not preclude adding it, but isn't asked to implement it.
- [ ] **No** — custom renderers are a core feature we ship ourselves, not user-extensible.

Default if unanswered: "Yes, later".

## Q3. What is the committed event surface for v1? (ref: `03-unbiased-approach.md` §5, `04-gap-analysis.md` §5)

The spike implemented `before_agent_start`, `tool_result`, and `registerTool`. Beyond these, pick which are worth adding now vs deferring:

| Event / capability | In v1? | Rationale needed if "yes" |
|---|---|---|
| `tool_call` (pre-execution, allow mutation + block) | ☐ | path-guard genre |
| `registerCommand` (slash commands) | ☐ | M9 scope overlap |
| `registerMessageRenderer` | ☐ | requires Q2 yes |
| `registerProvider` | ☐ | freezes model plumbing |
| `session_before_compact` | ☐ | M7 + M8 interaction |
| `session_start` / `session_shutdown` | ☐ | state hydration genre |

Default if unanswered: add none, stay at the spike's three.

## Q4. Trust model — what happens when a user enables an extension? (ref: `01-feasibility.md` §2 Axis III, `04-gap-analysis.md` §2)

- [ ] **No permission prompt** — built-ins are implicitly trusted. Fine only if Q1 = "Only us".
- [ ] **Trust-on-install** — first enable shows a dialog listing declared permissions; never again.
- [ ] **Trust-per-call** — a dialog for every new capability actually exercised.
- [ ] **Capability-based** — no permission strings; the extension only gets the objects we hand it.

Default if unanswered: tied to Q1. If "only us" → no prompt. If users involved → trust-on-install.

---

## Q5. What permissions are actually enforceable in v1? (ref: `04-gap-analysis.md` §2)

The spike declared three permission kinds and enforced one. Pick what must be enforced in v1:

- [ ] `net:<origin>` — already in spike.
- [ ] `fs:vault` — read? write? both? path-scoped within `/vault/`?
- [ ] `fs:extension` — per-extension key-value storage (scoped IDB partition).
- [ ] Tool invocation whitelist — extension may only call tools in its manifest.
- [ ] CPU budget — terminate a handler after N ms.
- [ ] Call count budget — throttle events dispatched to the extension.

Default if unanswered: `net:<origin>` + `fs:extension` only. Others deferred.

---

## Q6. Where does extension code execute? (ref: `01-feasibility.md` §2 Axis I)

Depends on Q1 + Q4. If user-authored untrusted code is in scope, Worker or iframe isolation is required. If only first-party, inline execution is far cheaper.

- [ ] **Inline in agent Worker** — shares lifetime with agent loop.
- [ ] **One shared user-extensions Worker** — one Worker total, all user-authored code inside, crash-isolated from agent.
- [ ] **One Worker per extension** — the spike's choice.
- [ ] **iframe sandbox** — only if Q2 = yes.

Default if unanswered: inline. Escalate only when Q1/Q4 force it.

---

## Q7. Distribution model for user-authored extensions, if any (ref: `01-feasibility.md` §2 Axis II)

Only relevant if Q1 includes user authoring.

- [ ] **Upload file (user picks a `.js` file).** Simplest UX.
- [ ] **Paste text (user pastes ESM into a textarea).** What the spike supports.
- [ ] **Pull from URL (user pastes a URL to a bundle).** Remote fetch + CORS trouble.
- [ ] **Sync from GitHub gist / repo.** Nicer UX, bigger scope.

Default if unanswered: upload file. Paste is still supported as a secondary path, URL is deferred.

---

## Q8. Authoring DX for users writing code (ref: `05-lessons-learned.md` §3.1)

- [ ] In-browser editor (codemirror + types). Best UX, biggest build.
- [ ] External editor + file upload. Cheap; expects the user to have a toolchain.
- [ ] No local authoring in v1; users only install extensions we ship.

Default if unanswered: tied to Q1. If only-us → last option. If users → external editor + upload.

---

## Q9. Lifecycle — how does enable/disable feel? (ref: `01-feasibility.md` §2 Axis IV, `04-gap-analysis.md` §1.1)

- [ ] **Rebuild required.** v1-built-in only.
- [ ] **Page reload on toggle.** Simplest; good UX with a clear confirm.
- [ ] **Agent-Worker restart on toggle.** No page reload; active turn lost.
- [ ] **True hot-swap.** The spike's goal; not reliably working.

Default if unanswered: "page reload" as the guaranteed path, "hot-swap" as an optimisation we earn with tests.

---

## Q10. Cross-tab consistency (ref: `04-gap-analysis.md` §1.1)

- [ ] **Not modelled.** Toggling in tab A is invisible in tab B until B reloads.
- [ ] **Soft sync.** A `BroadcastChannel` broadcasts the new enabled set; each tab picks it up at its next `agent_end`.
- [ ] **Strict sync.** Toggling in tab A forces tab B to reload.

Default if unanswered: soft sync.

---

## Q11. Where do extensions live in the UI? (ref: `04-gap-analysis.md` §3.5)

- [ ] **Popover in chat toolbar** — the spike's place.
- [ ] **Settings page** — alongside MCP, model config.
- [ ] **Both** — popover is quick toggle, settings page is full management.

Default if unanswered: settings page as canonical, optional popover for quick toggle.

---

## Q12. Naming (ref: `03-unbiased-approach.md` §9)

"Extensions" is ambiguous in our product. Does the user-facing term stay as "extensions"?

- [ ] **Extensions.** Matches what other tools call it.
- [ ] **Behaviours.** Accurate for v1 (built-in toggles).
- [ ] **Add-ons** / **Packs** / **Tweaks.** Softer; opens space for both toggle-style and code-style later.

Not a technical question; affects the milestone doc and every piece of copy.

---

## Q13. How does this relate to `coding-agent`? (ref: `05-lessons-learned.md` §2.2)

- [ ] **Port the `ExtensionAPI` as closely as we can** so extensions are shareable between the two tools.
- [ ] **Derive a smaller API from our target genres.** No cross-compatibility commitment.

This is the question that silently got answered "yes, port" in the spike. Worth revisiting.

---

## Q14. What ends the next iteration? (ref: all reports)

The spike ended with e2e tests passing. Those tests did not prove the feature works (§3.3 of lessons). What does "done" look like for the next iteration?

- [ ] A named list of real extensions we ship, each with a named user benefit.
- [ ] A gate that includes a build-preview e2e run.
- [ ] A user-observable trip-switch: a UI affordance to disable all extensions and verify the app still works.
- [ ] Something else.

Default: all three.
