# web-acp vs standard ACP — M2 divergence snapshot

> One-pager capturing the M2 divergences from the canonical
> Agent-Client Protocol posture. Read alongside

## Where web-acp matches standard ACP after M2

| Area | ACP canonical | web-acp M2 |
|---|---|---|
| `initialize` handshake | client advertises `clientCapabilities.fs.readTextFile/writeTextFile`; agent reports `protocolVersion` | **matches** — both flip to `true` in M2.3 |
| `session/new` / `session/load` / `session/prompt` | stable methods, stable shapes | **matches** |
| `session/update` | `agent_message_chunk`, `tool_call`, `tool_call_update` | **matches** — `bash` tool uses the same wire |
| `session/cancel` → tool abort | cooperative cancel, tool reports `cancelled` | **matches** — per-turn `AbortController` propagates to `bash.exec({ signal })` |
| Tool-call shape | `kind`, `rawInput`, `rawOutput`, `status`, `content` | **matches** — bash reports `kind: 'execute'` |

## Where web-acp diverges from standard ACP

### 1. Agent owns the FS; `fs/*` is an IDE-integration seam only

**Standard ACP.** `fs/readTextFile` + `fs/writeTextFile` are the
editor-buffer bridge. Thick-agent reference impls (e.g.
`claude-agent-acp`) route tool-driven reads/writes through these
methods so the editor can intercept for unsaved-buffer semantics.

**web-acp M2.** The built-in `bash` tool **never calls `fs/*`**.
just-bash has a 25-method `IFileSystem` (stat, readlink, symlink,
rename, etc.); ACP `fs/*` has two methods. Round-tripping the
richer surface through the thinner one would lose capability and
add cost. Worker-side ZenFS (`WebAccess` over FSA handles) is the
agent's authoritative FS. `fs/*` handlers still ship on the main
thread — advertised in `clientCapabilities` — as a seam for
external ACP agents or a future editor-buffer bridge.

**Compliance posture.** Compliant on the wire (handlers exist
and respond correctly); divergent on the intended usage pattern
(built-ins don't call their own client's `fs/*`). Documented in
[`../specs/web-acp/vault.md`](../specs/web-acp/vault.md) §
"IDE-integration seam".

### 2. Volume mounts advertised in the system prompt, not via a capability

**Standard ACP.** No canonical way to advertise FS layout to the
agent; the agent is expected to discover it via `fs/*` or via
out-of-band config.

**web-acp M2.** Multi-volume mounts live at `/mnt/<name>` and
each volume carries an optional description. The worker composes
a `Volumes:\n- /mnt/<name> — <description>\n…` block into the
system prompt so the LLM knows what's in each mount. Future
work may lift this into a structured extension method, but M2
keeps it inline.

### 3. `_bodhi/features/*` extension methods

**Standard ACP.** No generic feature-toggle surface.

**web-acp M2.** Two new `_bodhi/*` extension methods:
`_bodhi/features/list` returns `{ features, defaults }`;
`_bodhi/features/set` writes one flag. `bashEnabled` (public,
default on) and `forceToolCall` (DEV-only, default off) ship in
M2. The surface is generic — M3 MCP toggles and later
native-tool enables will layer on without reshaping it.

**Compliance posture.** Extension surface, per principle 15.
Not a divergence from ACP; an additive wire on top of it.

### 4. Permission bridge deferred

**Standard ACP.** `session/request_permission` is a stable
method; canonical agents gate destructive tool calls on it.

**web-acp M2.** No `session/request_permission` calls from the
built-in `bash` tool. Documented on
[`../milestones/deferred.md`](../milestones/deferred.md) with a
re-entry ticket and the prerequisites already satisfied
(`bash` registered, `session/cancel` wired,
volumes advertised).

**Compliance posture.** Non-compliant on the "destructive-
command consent" expectation implied by ACP's permission
primitives. The compliance table in
[`../milestones/index.md`](../milestones/index.md) carries the
row as `deferred (see deferred.md)`.

### 5. Main-thread duplicate ZenFS (supporting the `fs/*` seam)

**Standard ACP.** Client-side `fs/*` handlers typically wrap the
editor's open-buffer set or the host filesystem.

**web-acp M2.** The main thread keeps a **duplicate** ZenFS
context that mounts the same `FileSystemDirectoryHandle`s the
worker uses, so `fs/*` handlers can read/write the same bytes
without round-tripping through the worker. FSA handles are
structured-cloneable and the OS storage is shared; writes from
one side are visible to the other for real volumes. In-memory
seed volumes are independently seeded on each side (dev/test
only).

**Caveat.** Writes from the worker's `bash` tool into a
seed-mode volume are not visible to main-thread `fs/*`
handlers. Tests stage fixtures on both sides explicitly.
Coordination for production-mode concurrent writes is deferred
and documented in
[`../specs/web-acp/vault.md`](../specs/web-acp/vault.md).

## Summary

web-acp M2 matches the ACP wire on every shipped surface and
advertises the standard `fs/*` capability bit honestly. The
strategic divergence is that the built-in tool doesn't *use*
`fs/*` — the agent-owned FS is richer than the client-owned
one, so the worker keeps authority and the main thread plays
IDE-integration seam. The deferred permission bridge is the
only non-compliant row; re-entry criteria are documented and
the tool-call wire is already shaped to support it.
