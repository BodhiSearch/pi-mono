# M3.5 — Post-M3 follow-ups

**Status:** shipped. Two small follow-ups landed after the M3 exit
gate that don't reshape M3's MCP wire but extend the surface in
ways worth recording before M4 picks up.

## What this milestone delivered

Two independent changes, both shipped between the M3 exit gate
(`6be4782a`) and the M4 kickoff:

1. **Multi-MCP-server login flow** (commit `2ef309ce`) — the login
   builder now requests scope for a second public MCP server
   alongside the existing everything-server fixture URL.
2. **Persistent session deletion** (commit `b0d136c8`) — a new
   `_bodhi/sessions/delete` extension method plus per-row delete
   affordance in the session picker. M1 shipped create / list /
   load; this closes the lifecycle.

## Sub-milestones as shipped

### M3.5.1 — DeepWiki MCP scope on login

Source surface:

- `packages/web-acp/src/components/Header.tsx` — adds
  `resolveDeepwikiMcpUrl()` alongside the existing
  `resolveEverythingMcpUrl()`. Default URL is the public
  `https://mcp.deepwiki.com/mcp`; honours an optional
  `VITE_MCP_DEEPWIKI_URL` override (empty string disables so
  deployments can opt out or pin to a self-hosted mirror).
- `handleLogin` invokes `LoginOptionsBuilder.addMcpServer(...)`
  twice — once per resolved URL — before building the login
  options.

Why it matters. Establishes the **multi-server login pattern**:
the Bodhi access-request flow accepts an arbitrary list of
upstream MCP URLs to mediate, and the M3 catalog plumbing
(`bodhiClient.mcps.list()` →
[`useMcpInstances`](../specs/web-acp/mcp.md) → composed
`McpServerHttp[]`) already handles N servers without churn. No
agent or worker code changed.

### M3.5.2 — `_bodhi/sessions/delete`

ACP surface added:

- `_bodhi/sessions/delete` — agent-side extension method
  (client → agent). ACP has no stable `session/delete`;
  `session/close` in the unstable schema means "free in-memory
  resources", not "remove from disk", so per principle § 15 we
  ride a `_bodhi/*` namespace.

Source surface:

- `packages/web-acp/src/acp/index.ts` — `BODHI_SESSIONS_DELETE_METHOD`
  constant, `BodhiSessionsDeleteRequest` /
  `BodhiSessionsDeleteResponse` types.
  `deleted: false` in the response means the worker had no record
  of the session — the call is treated as a no-op rather than an
  error so clients can issue delete-after-delete (or race two
  delete clicks) without surfacing a spurious failure.
- `packages/web-acp/src/acp/agent-adapter.ts` — handler releases
  pooled MCP connections, drops the in-memory session state,
  clears the inline-agent buffer if the session was active, then
  calls the existing atomic `store.deleteSession` (entries /
  features / `mcpToggles` cascade in one Dexie transaction).
- `packages/web-acp/src/acp/client.ts` — `deleteSession()`
  wrapper.
- `packages/web-acp/src/hooks/useAcp.ts` — `deleteSession`
  callback. Active-session case: cancels the in-flight prompt
  first, then runs `clearMessages()`; the existing auto-create
  effect provisions a fresh session.
- `packages/web-acp/src/components/chat/SessionPicker.tsx` —
  per-row delete affordance. UX agreement: single click, no
  confirmation.
- `packages/web-acp/src/components/chat/ChatDemo.tsx` —
  threads `deleteSession` through to the picker.

### Test inventory

- `packages/web-acp/src/acp/agent-adapter.test.ts` — three new
  vitest cases: happy path, idempotent unknown-id, malformed
  params.
- `packages/web-acp/e2e/sessions-delete.spec.ts` — Playwright
  spec that creates two sessions, deletes the inactive one,
  deletes the active one, and confirms the freshly auto-created
  replacement session is usable for a follow-up prompt.
- `packages/web-acp/e2e/tests/pages/ChatPage.ts` — picker page
  object gains a `deleteSession(id)` helper.

## Decision log

- **Multi-server login is environment composition, not
  protocol churn.** Adding a second `addMcpServer(...)` call is a
  Bodhi access-request affordance; the agent-side MCP wire (M3)
  treats the resulting `McpServerHttp[]` uniformly. No
  agent-adapter changes were needed.
- **Delete rides `_bodhi/sessions/*`, not `session/close`.** ACP's
  unstable `session/close` semantics are about freeing in-memory
  resources after a turn; persistent removal is an agent-side
  concern. The `_bodhi/*` extension keeps M3.5 strictly
  ACP-canonical-or-extension and follows principle § 15.
- **Idempotent delete (`deleted: false` for unknown ids).**
  Avoids a noisy error path for the double-click race and lets
  clients call delete optimistically without first looking up
  whether the session is still on disk.
- **Active-session cleanup is a hook concern, not adapter
  concern.** The worker handler always clears the active inline
  state if the deleted session matches it; the hook is
  responsible for cancelling the in-flight prompt and clearing
  the visible transcript before the auto-create effect fires.
  Splitting the responsibility means the worker stays a single
  source of truth for persisted state, while the hook owns
  in-flight UI side-effects.

## Out of scope (this milestone)

- A confirmation modal on delete. Single click was the explicit
  product call.
- A "trash" / undo flow. Deletion is destructive and immediate.
- Cross-session bulk delete (e.g. "delete all sessions older than
  X"). Per-row only.
- Promoting `_bodhi/sessions/delete` to the ACP spec. If
  upstream lands a stable `session/delete`, principle § 15
  carries the migration path (advertise both for one milestone,
  switch, remove `_bodhi/*`).

## Why this ordering

Both items are small, independent, and post-exit-gate. Folding
them into the M3 doc would muddy the M3 exit-gate snapshot;
holding them for M4 would make the M4 diff carry unrelated
context. A dedicated M3.5 file keeps the trail clean.
