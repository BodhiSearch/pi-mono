## Context — where we are

M2 shipped the agent-owned FS and a single `bash` tool. The tool
registry is ready for additional entries — `pi-agent-core`'s
`AgentTool` interface is our in-house contract, and the
`_bodhi/features/*` toggle surface gives us a home for per-tool
enables. Session persistence (M1) already carries a generic
`features` bag; `bodhi/getSession` snapshots it for rehydration.

M3 doesn't reshape any of that. It adds a catalog mechanism
(MCP) and a passthrough mechanism (provider-native). Both land
as ordinary `tool_call` / `tool_call_update` notifications on
the existing wire — the UI rendering pattern from M2's
`BashToolCall.tsx` generalises.

## Decisions already made (do not re-ask these)

1. **Agent is the MCP client.** The worker owns MCP connections;
   the main thread only renders settings and pushes server config.
   Matches ACP's canonical posture
   (`agent-client-protocol/docs/protocol/session-setup.mdx`).
2. **HTTP transport only for v1.** Streamable HTTP per the MCP
   spec. No stdio (browsers can't spawn processes), no SSE
   (deprecated upstream).
3. **Tool namespacing: `<serverName>__<toolName>`.** Avoids
   collisions with `bash` and between servers. Double-underscore
   keeps names regex-friendly for routing.
4. **No permission prompts in M3.** MCP tools run as-is, same as
   `bash`. The permission bridge remains deferred; it re-enters
   later and will cover both uniformly.
5. **Provider-native tools are passthrough.** The agent enables
   them per-session via provider config, observes the provider's
   SSE tool events, and re-emits ACP `tool_call` +
   `tool_call_update` notifications so the UI renders them
   consistently. The agent never dispatches them.
6. **Server config persists app-wide, connections do not.**
   Server list sits in main-thread IndexedDB; worker reconnects
   on boot and when the list changes. One session doesn't "own"
   a server list — all sessions share the configured servers.
7. **Extension methods prefixed `_bodhi/*`.** New methods:
   `_bodhi/mcp/setServers` (client → agent),
   `_bodhi/providers/nativeTools` (client → agent). Constants
   live in `acp/methods.ts` per principle 15.
8. **Native-tool toggles are per-session per-model.** Stored in
   the session record's `nativeTools` bag, rehydrated by
   `bodhi/getSession` on reload.

## Deliverables

Every deliverable lands inside the phase commit that produces
it. Spec updates share the commit with the code that motivates

### Phase A — M3.1 — MCP HTTP client (worker-side)

- Settings UI on the main thread for configuring MCP servers
  (name, URL, auth header template). Persist in main-thread
  IndexedDB via `idb-keyval` (same pattern as
  `vault/fsa-handle-store.ts`). No ZenFS involvement.
- `_bodhi/mcp/setServers` extension method (client → agent).
  Constants land in `acp/methods.ts`. Worker persists the list
  in Dexie (new `mcpServers` table, version bump), reconnects
  as needed.
- Worker-side MCP HTTP client: official
  `@modelcontextprotocol/sdk` browser client if it ships by
  implementation time, otherwise a minimal hand-rolled
  Streamable HTTP client. Discover tools via `tools/list`;
  register namespaced `<serverName>__<toolName>` entries with
  `pi-agent-core`'s registry alongside `bash`.
- `agentCapabilities.mcpCapabilities.http = true` advertised in
  `initialize` response.
- UI: `data-testid="mcp-panel"` with per-server rows
  (`data-testid="mcp-server-<name>"`,
  `data-test-state="disconnected|connecting|connected|error"`).
- Spec updates: new
  [`../specs/web-acp/mcp.md`](../specs/web-acp/mcp.md)
  documenting server-list shape, transport choice,
  reconnection policy, tool namespacing, and the
  `_bodhi/mcp/*` wire.

**Phase A gate:** `npm run check` + vitest + full M2 e2e +
new `mcp-connect.spec.ts` pointing at a published HTTP-transport
test server (or a locally spawned fixture in `e2e/fixtures/`).
Commit `web-acp: M3 phase A — MCP HTTP client (worker-side)`.

### Phase B — M3.2 — MCP tool invocation through the unified registry

- Route tool calls by name: `bash` → just-bash; `srv__tool` →
  the matching MCP client; unknown → structured error.
- Wrap `tools/call` with a `session/cancel`-aware
  `AbortController`; stream server-side progress as
  `tool_call_update` with `rawInput` / `rawOutput` preserved.
- MCP-side error envelopes translate into ACP tool-call errors
  with the server's error text preserved (same `failed` status
  as `bash` errors).
- **No permission prompts.** Document the posture in
  `mcp.md` + cross-reference `deferred.md`.
- Vitest: MCP client stub + registry routing tests; failure
  translation; abort plumbing.
- e2e: real MCP server round-trip (calculator fixture); mixed
  `bash` + MCP call within one turn.

**Phase B gate:** `npm run check` + vitest + full previous
e2e suite + `mcp-roundtrip.spec.ts`. Commit
`web-acp: M3 phase B — MCP tool invocation via unified registry`.

### Phase C — M3.3 — Provider-native tool passthrough

- `_bodhi/providers/nativeTools` extension method returning the
  per-model native-tool catalog the current provider supports
  (derived from `pi-ai` capability metadata).
- Settings UI: per-model toggles
  (`data-testid="native-tool-<model>-<tool>"` with
  `data-test-state="on|off"`). Defaults off.
- Session record gains `nativeTools: Record<modelId, Record<toolName, boolean>>`.
  Persists + rehydrates via `bodhi/getSession`.
- Adapter: when a user enables a native tool for a model and
  runs a prompt against that model, pass the tool config to
  `pi-ai`; observe the provider's SSE tool events; re-emit as
  ACP `tool_call` + `tool_call_update` (preserving tool name,
  arguments, and output).
- Native tools never dispatch through the agent — observability
  only. Document that toggle-to-enable is the user consent for
  these tools (the provider's own UX handles prompting if
  relevant).
- Spec updates: new
  [`../specs/web-acp/native-tools.md`](../specs/web-acp/native-tools.md);
  edits to `sessions.md` (new slot), `features.md`
  (relationship to the generic feature surface — native tools
  are a separate bag because they are model-scoped, not
  boolean-scoped).

**Phase C gate:** `npm run check` + vitest + full previous
e2e + `native-tools.spec.ts` (enable OpenAI `web_search`, ask
a time-sensitive question, assert the `tool_call` with
`kind: 'search'` surfaces; disable → no `tool_call` on the
same question). Commit
`web-acp: M3 phase C — provider-native tool passthrough`.
