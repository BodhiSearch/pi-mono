## Context — where we are

M0 shipped ACP framing over a `MessageChannel`-backed Web
Worker. The agent lives inside the worker as `AcpAgentAdapter`
wrapping `InlineAgent` (a thin shim over `pi-agent-core`'s
`Agent`). The main thread owns an `AcpClient` via the
`useAcp` hook. M0 left no persistence: every reload drops the
session.

The reference worker-agent spike at
`ai-docs/specs/worker-agent/sessions.md` +
`agent-session.md` solved the same problem with bespoke RPC.
M1 rebuilds that capability on top of ACP — the agent owns
session state, the client addresses it by `sessionId`, and
restoration happens via the stable `session/load` method.

## Decisions already made (do not re-ask these)

1. **Resume path is ACP stable `session/load`.** Not an
   app-level replay. Justification: ACP's stable schema already
   defines `LoadSessionRequest` / `LoadSessionResponse`;
   inventing our own replay violates principle 2 ("ACP is the
   only internal protocol"). pi-acp's agent implements it
   identically — use it as a shape reference, not a
   dependency.
2. **Session store is worker-owned.** `SessionStore` lives
   inside the worker and is constructed by `agent-worker.ts`,
   then passed into `AcpAgentAdapter`. The main thread only
   ever sees session data through ACP. This keeps the agent
   authoritative and makes the remote-agent transport a trivial
   swap later.
3. **Listing goes through a Bodhi ext method
   (`bodhi/listSessions`).** The SDK's `listSessions` is only
   declared in `schema.unstable.json`; M0 committed to stable
   schema only. When upstream stabilises we rename the constant
   and keep the wire shape compatible.
4. **Replay re-emits stored `SessionNotification`s verbatim.**
   This matches ACP's spec: "stream the entire conversation
   history back to the client via notifications" in
   `schema.json § LoadSessionRequest`.
5. **Model restoration rides a companion snapshot call.**
   `session/load` has no first-class place for "last selected
   model" in its stable response. Add a second Bodhi ext
   method, `bodhi/getSession`, that returns a collapsed
   snapshot (`{messages, lastModelId, title}`). The hook calls
   both — `session/load` for protocol fidelity,
   `bodhi/getSession` for one-shot UI rehydration. Reference:
   `packages/coding-agent/` stores the model per session in
   the session row; we follow the same shape.
6. **No `schemaVersion` column.** ACP does not define one at
   store granularity. Inventing one now pretends to solve
   forward-compat without buying us anything; address
   migrations in the milestone that actually needs them.
7. **M0 hardening items (second transport, worker-boundary
   e2e) are deferred.** Not part of M1 per user decision.
8. **Two-model e2e coverage.** The `.env.test` provisions
   both an OpenAI model and an Anthropic model, mirroring
   `packages/web-agent/e2e/.env.test`. One session per model;
   switching between them in the resume e2e proves the model
   selector is rehydrated per session.

  `[sessionId+seq]`), CRUD, and turn aggregation. `entries`
  carries two kinds: `'notification'` (raw

  `restoreMessages(messages)` and `getMessages()` for the

  optional `SessionStore`; `newSession` calls
  `store.createSession` and resets the inline runtime via a
  new `#activeInlineSessionId` tracker; `#emit` helper
  persists alongside `conn.sessionUpdate`; `prompt` writes a
  `turn` row on clean `end_turn`.
- `packages/web-acp/src/agent/agent-worker.ts` — instantiates
  the store and passes it to the adapter.
- `packages/web-acp/src/agent/session-store.test.ts` — vitest
  coverage (create → append notifications → append turn;
  monotonic seq; title derivation; delete).
- Spec updates: new
  [`../specs/web-acp/sessions.md`](../specs/web-acp/sessions.md);
  edits to `index.md` (nav + layout), `agent.md` (inline-agent

- Deps: `dexie@^4` + `fake-indexeddb` (dev) added to

### Phase C — Resume sessions (`session/load` + switcher)

- `packages/web-acp/src/acp/agent-adapter.ts`:
  - `initialize` advertises `agentCapabilities.loadSession: true`
    when a store is wired in.
  - `loadSession` reads entries, re-emits notifications
    verbatim (bypassing `#emit` so replays don't double-persist),
    seeds `InlineAgent` from the last `turn.finalMessages`,
    sets `#activeInlineSessionId`.
  - `bodhi/getSession` ext method returns `{sessionId,
    messages, lastModelId, title}` as the collapsed snapshot.
  - `prompt` adds a mismatch guard — if the incoming
    `sessionId` differs from `#activeInlineSessionId`, it
    rehydrates the inline runtime from the store before
    running the turn. Prevents one session's history from
    contaminating another's `finalMessages`.
- `packages/web-acp/src/acp/client.ts` — `loadSession(id)` +
  `getSession(id)` wrappers.
- `packages/web-acp/src/hooks/useAcp.ts`:
  - `currentSessionId`, `isLoadingSession` state.
  - `isReplayingRef` to silence the live `session/update`
    listener during replay.
  - `loadSession(id)` callback: calls `client.loadSession`
    then `client.getSession`, sets messages in one shot,
    restores `selectedModel` / `selectedApiFormat` from
    `lastModelId`.

- `packages/web-acp/src/components/chat/SessionPicker.tsx` —
