## Context — where we are, why we are pivoting

### What `packages/web-agent/` delivered

`web-agent` is a browser-native coding-agent harness. It shipped M0–M8
(see `ai-docs/milestones/index.md`):

- **M0–M3** Workspace integration, RPC scaffold, ZenFS `/vault` mount
  via Chrome File System Access, 6 filesystem tools.
- **M4** Worker transport — `AgentSession` runs inside a Web Worker,
  `rpc-client` ↔ `rpc-server` over `MessageChannel`, a second
  `MessageChannel` tunnels ZenFS ops for vault I/O.
- **M5** Session persistence on Dexie/IndexedDB; `session_loaded`
  event; explicit `SessionStore` interface.
- **M6** Session tree — `forkSession`, branch navigation, per-message
  Fork/Branch UI.
- **M7** Compaction — auto + manual, summary bubble with
  `data-test-state` hooks.
- **M8** Extensions runtime (Phases 1 + 2a + 2b): `.pi/extensions/<name>/index.js`
  loaded via Blob-URL `import()`, every context/lifecycle hook,
  full `pi.*` registration surface, modal `pi.ui.*` channel, readonly
  session forwarder, compaction hooks. **Phase 3 isolation deferred
  by product decision — extensions are fully trusted** (see
  `ai-docs/milestones/deferred.md`).

Authoritative tech reference: `ai-docs/specs/worker-agent/` (module
specs) and `ai-docs/milestones/` (status board + outcome summaries).

### Why `web-agent` does not become the v1

`web-agent` works, but along the way the boundary between **client
responsibilities** (main thread, UI, vault handle, user input) and
**agent responsibilities** (session state, LLM calls, tools,
extensions, compaction) got blurred in several places:

- **Dual-responsibility RPC.** `rpc-types.ts` is a bespoke protocol
  shaped for our UI rather than the agent. New UI features force
  protocol churn; there is no separation between "what the agent
  exposes" and "what the UI binds to".
- **Vault lifecycle split across threads.** The main thread owns the
  FSA handle and the ZenFS mount; the worker re-reaches through a
  second `MessageChannel` for every `ls`/`read`/`write`. Works, but
  every tool ends up with a main-thread round-trip that wasn't in the
  original `coding-agent` shape.
- **Extension UI is co-designed with the chat UI.** `ExtensionUIController`
  on the worker side drives `ExtensionUIRenderer` on the main side via
  a parallel RPC channel (`extension_ui_request` / `extension_ui_response`).
  The result is two protocols where ACP would give us one.
- **Slash commands / skills / prompts** are a mix of builtin
  (registered on the main thread through `useAgent.intercept`),
  worker-scanned (`CommandRegistry`), and worker-sandboxed (bash shim
  inside an iframe + Worker). Discovery paths, precedence rules, and
  error surfacing differ across the three sources.
- **Extensions sandboxing** consumed a full phase (M8 Phase 3) to
  decide, and only closed by *dropping* the isolation requirement
  entirely. A cleaner architecture would have made the trust question
  structural, not an afterthought.

None of these are bugs. They are the natural consequence of growing
the wire protocol outwards from the UI instead of inwards from a
stable contract.

### The bet

**ACP is that stable contract.** It is designed for exactly this
split: a dumb client that shows rich UI, a rich agent that streams
events, with filesystem and terminal delegation flowing the other way.
If we build `web-acp` around ACP from day one:

- The client only speaks ACP. No bespoke RPC.
- The agent only speaks ACP. No UI-shaped protocol.
- Filesystem / permission / auth delegation are protocol primitives,
  not our invention.
- Every future feature (resources, extensions, themes, compaction) is
  "how does this map onto ACP?" instead of "invent a new RPC verb".

`coding-agent` already speaks ACP via `pi-acp`
(https://github.com/svkozak/pi-acp). We get the protocol for free, and
we get a reference implementation that proves the ergonomics.

## Decisions already made (do not re-ask these)

The invoker has already decided the following. Your job is to **work
within** these, not relitigate them.

1. **Package location.** `packages/web-acp/` is already bootstrapped
   via `create-bodhi-js`. It sits alongside `packages/web-agent/` in
   the monorepo. Treat it as the active package going forward. Wire
   it into any root-level scripts that make sense (`npm run check`
   already covers web-ui + web-agent — extend to web-acp when we start
   landing real code, not this turn).
2. **ACP transport target.** **Worker-only.** Main thread = ACP
   client. Web Worker = ACP agent. ACP JSON-RPC 2.0 is framed over a
   `MessageChannel`. No network hop. A remote-agent transport is
   *potentially* a future milestone, but every design call for v1
   should assume in-browser only. Keep the transport layer swappable
   though — don't hardcode `MessagePort` in the ACP framing code.
3. **web-agent reuse policy. Reference-only.** Do not copy files
   across from `web-agent` into `web-acp`. Re-derive everything
   deliberately. You may and should **read** web-agent's specs
   (`ai-docs/specs/worker-agent/`), its git log, its e2e tests, and
   its own post-milestone reports — every one of them is a crib
   sheet. But every line of code in `web-acp` is written fresh with
   the ACP contract in mind.
4. **Extensions scope.** **Late milestone.** v1 of `web-acp` ships
   without any extension runtime. The "fully trusted extensions +
   Blob-URL loader" decision from M8 Phase 3 is a sunk cost of
   `web-agent`, not a starting point for `web-acp`. When extensions
   re-enter (after persistence / compaction / resources are solid),
   the design starts from **"how does ACP extend?"** — not from the
   existing loader.
5. **M0 e2e fidelity.** **Real LLM call** for the M0 "mount folder +
   ask a question" test. Copy the `.env.test` from
   `packages/web-agent/e2e/.env.test` (or its `.example` equivalent —
   verify which is committed) and use the same provider the existing
   e2e uses. DOM-witness assertions only, no LLM-text assertions.
   Same discipline as `web-agent/e2e/`.
