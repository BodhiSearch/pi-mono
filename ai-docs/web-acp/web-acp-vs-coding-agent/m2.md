# web-acp vs coding-agent — M2 posture snapshot

> One-pager contrasting web-acp's **single-`bash`** tool surface
> (M2) with the richer, multi-tool surface of the in-monorepo
> `packages/coding-agent/`. Read alongside
> [`../milestones/m2-tools.md`](../milestones/m2-tools.md) and
> [`../specs/web-acp/tools.md`](../specs/web-acp/tools.md).

## Tool catalogs

| Aspect | coding-agent | web-acp M2 |
|---|---|---|
| Tool count | many (read-file, write-file, glob, grep, shell, edit, apply-patch, …) | **one**: `bash` |
| Shell surface | `ShellTool` wrapping a Node child process | `bash` wrapping `just-bash` in a Web Worker |
| File reads | dedicated `ReadTool` with line windowing | `bash`'s `cat` / `sed -n` / `head` / `tail` |
| Search | dedicated `GrepTool`, `GlobTool` | `bash`'s `grep`, `rg`, `find` |
| Patching | dedicated `ApplyPatchTool` with structured diffs | `bash`'s `sed`, redirect writes |
| Filesystem auth | OS process inherits host permissions | FSA handles + persisted via `idb-keyval` + re-grant flow |
| Sandbox | OS process boundary | browser sandbox + just-bash's threat model |
| MCP integration | present (coding-agent hosts MCP clients today) | **M3** (deferred to next milestone) |

## Why the single-tool surface for M2

1. **just-bash bundles the primitives.** `cat`, `ls`, `grep`,
   `rg`, `find`, `sed`, `awk`, `jq`, pipes, redirects — all ship
   as one dependency. Splitting them into individual ACP tools
   would reinvent shell composition at the wire layer.
2. **Composition is the LLM's job.** Modern models compose shell
   pipelines well; a single `bash` tool lets the LLM express
   `ls /mnt/wiki | grep markdown | head -5` in one call instead
   of three round-trips. Fewer `tool_call` notifications per
   turn reduces latency and token cost.
3. **One permission gate to design, later.** When the permission
   bridge re-enters (deferred), gating a single tool with a
   classifier plugin is simpler than per-tool UX for a dozen
   tools. The `BashTransformPipeline` seam in just-bash is the
   natural landing spot.
4. **Browser ≠ OS.** coding-agent's tools assume a host
   filesystem; web-acp runs under FSA handles with no
   `chmod`, no symlinks from the host, no process spawn. A
   single bash tool hides those differences behind POSIX-ish
   behaviour the LLM already knows.

## Where the shapes still converge

- **Streaming.** Both agents emit incremental output (coding-
  agent: per-tool; web-acp: ACP `tool_call_update` with
  `content: { type: 'content', content: [{ type: 'text', text }] }`).
- **Abort.** Both use `AbortController`s driven by user cancel.
- **Structured output.** coding-agent's `ShellTool` returns
  `{ stdout, stderr, exitCode }`; web-acp's `bash` returns the
  same shape plus `{ truncated?: boolean }`.
- **Tool registry.** Both register tools in a runtime registry
  and let the LLM drive selection.

## Implications for future milestones

- **M3 (MCP + native tools).** web-acp will register MCP tools
  namespaced as `<serverName>__<toolName>` alongside `bash`.
  coding-agent already does the equivalent; we can port the
  routing pattern (by name) but not the transport (coding-agent
  uses stdio where available; web-acp is HTTP-only).
- **Permission bridge.** Carried to a later milestone. When it
  lands, the classifier wraps `bash` specifically — MCP tools
  get a different, simpler gate (first-call-prompt) because
  the risk surface is different.
- **Patching tools.** coding-agent's `ApplyPatchTool` is a
  candidate for a future M4+ built-in if LLM-driven `sed` /
  `patch` flows prove error-prone under real traffic.

## Summary

web-acp M2 is deliberately narrower than coding-agent. One tool,
one permission surface (deferred), one FS adapter. The design
trades tool-catalog breadth for composability (via shell) and
simpler security reasoning (one gate to design). M3 onward
widens the catalog without changing this shape: MCP and
provider-native tools layer alongside `bash` on the unchanged
tool-call wire.
