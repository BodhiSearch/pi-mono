# packages/web-agent — browser-native coding-agent

## Context

`packages/web-agent` is a React + Vite app bootstrapped from `create-bodhi-js`. It already authenticates with a local BodhiApp server, lists Bodhi-hosted models, and streams chat responses through `pi-agent-core` + `pi-ai`, with MCP tools from the Bodhi backend. An e2e Playwright test covers login → model select → ask a question → assert reply.

We now want to evolve this app into a **browser-native coding agent** that operates on a user-picked local folder mounted through the Chrome File System Access API (FSA), abstracted by **ZenFS**. The agent will mirror `packages/coding-agent`'s architecture (agent loop + tool system + RPC + extensions) but run entirely in the browser. To keep web-agent portable and eventually extractable into its own package, all agent code will live under `packages/web-agent/src/web-agent/` and will not depend on `packages/coding-agent`; it will consume `packages/ai` and `packages/agent` plus `@bodhiapp/bodhi-js-react` for the LLM path.

Development is test-driven: minimal but high-coverage Playwright black-box tests with `test.step` + assertions, seeded via a dev-mode-only vault seam (same pattern `bodhiapps/zenfs-browser` uses) so tests don't need to click a user-gesture-gated picker.

## Scope of this plan

This plan covers **Phase 0** (workspace integration + Vite warnings) and **Phase 1** (agent-shaped scaffolding under `src/web-agent/`) in full detail, then sketches Phases 2–6 so we share a roadmap. Each subsequent phase will be re-planned before execution.

### 0.2 Fix Vite warnings from `packages/ai`

The warnings come from Vite's import analysis flagging two identifiers it cannot statically analyze:

- `packages/ai/src/env-api-keys.ts:8` — `const dynamicImport: DynamicImport = (specifier) => import(specifier);`
- `packages/ai/src/providers/register-builtins.ts:95` — `const importNodeOnlyProvider = (specifier: string): Promise<unknown> => import(specifier);`

Both are deliberate node-only lazy imports guarded by `typeof process !== "undefined" && process.versions?.node`. In the browser, they are never called. The warning is cosmetic; Vite's own message says *"If this is intended to be left as-is, you can use the `/* @vite-ignore */` comment inside the `import()` call to suppress this warning."*

**Fix:** edit both source lines to add the hint:

```ts
const dynamicImport: DynamicImport = (specifier) => import(/* @vite-ignore */ specifier);
// and
const importNodeOnlyProvider = (specifier: string): Promise<unknown> => import(/* @vite-ignore */ specifier);
```

Then rebuild `packages/ai` so the warning disappears from `dist/` too. If we're already depending on the workspace symlink (0.1), web-agent's dev server consumes `dist/` after rebuild; add `dev:tsc` watcher for `ai` during development (already present in root `dev:tsc` script).

## Phase 1 — scaffold `src/web-agent/` (agent shape, no filesystem yet)

**Goal:** create the directory layout that will host the RPC-based agent core, wire the existing `useAgent` hook to a thin in-process RPC client so the UI is already decoupled from the agent, and keep the existing chat e2e passing.

### 1.1 Directory layout (mirrors coding-agent)

```
packages/web-agent/src/web-agent/
├── core/
│   ├── agent-session.ts           # turn-loop wrapper around pi-agent-core
│   ├── session-state.ts           # in-memory session state types
│   ├── tools/
│   │   └── index.ts               # empty export for now; Phase 3 adds fs tools
│   └── extensions/
│       ├── types.ts               # hook/event types (copied shape from coding-agent/extensions/types.ts)
│       └── registry.ts            # static registration API (Phase 5 extends to dynamic)
├── rpc/
│   ├── rpc-types.ts               # commands/responses/events (shape from coding-agent/modes/rpc/rpc-types.ts)
│   ├── rpc-server.ts              # dispatches RpcCommand → AgentSession methods; emits events
│   ├── rpc-client.ts              # browser-side client with typed methods + event subscription
│   ├── transport.ts               # Transport interface (send, onMessage)
│   └── transports/
│       └── in-process.ts          # MessageChannel-based transport, Phase 4 adds worker transport
└── index.ts                       # barrel for the package-to-be
```

**Reuse from coding-agent (copy + trim, do NOT import):**
- `rpc-types.ts`: `RpcCommand`, `RpcResponse`, `RpcSessionState`, event envelope. Drop node-specific commands (`bash`, `fork`, session file import/export) for now; keep `prompt`, `abort`, `get_state`, `set_model`, `steer`, `follow_up`.
- `extensions/types.ts`: the event union (`BeforeAgentStartEvent`, `ToolCallEvent`, `ToolResultEvent`, `TurnEndEvent`, etc.) and `ExtensionContext`, `Extension`, `ExtensionFactory` types. Skip anything that references the jiti loader or node fs.

**Reuse from `packages/agent` directly (import):**
- `agentLoop` / `agentLoopContinue`, `AgentEvent`, `AgentTool`, `AgentMessage`, `StreamFn` — these are already browser-safe.

### 1.2 `AgentSession` (browser version)

Minimal wrapper around the existing `Agent` class from `@mariozechner/pi-agent-core`. Responsibilities:

- hold session state (messages, model, tools, system prompt, isStreaming, abort controller)
- expose methods: `prompt(text)`, `abort()`, `getState()`, `setModel(model)`, `setTools(tools)`, `steer(text)`, `followUp(text)`
- emit AgentEvents to a single subscriber callback (the RPC server forwards them)

No filesystem, no extensions yet; those are injected later via `AgentSessionServices` (dependency object constructed from outside).

### 1.3 RPC server + client

- **Transport-agnostic RPC core**: `RpcServer` takes a `Transport` (`{ send(msg), onMessage(cb) }`) and dispatches incoming `RpcCommand` by type to `AgentSession`. Responses flow back with the original `id`. Agent events are forwarded as unsolicited event messages.
- **In-process transport**: `MessageChannel` with two ports — one held by server, one by client. Lives on the main thread in Phase 1. Phase 4 will swap in a Web Worker without touching the dispatcher.
- **Typed client**: `RpcClient` exposes `prompt(text)`, `abort()`, `getState()`, etc. and a `subscribe(cb)` for events. All methods correlate by id with promise resolution.

### 1.4 Rewire the React layer

Replace the existing `hooks/useAgent.ts` internals with a thin adapter over `RpcClient`:

- `useAgent(tools)` constructs an `RpcServer` + in-process transport + `RpcClient` once per mount (module-level singleton, matching the current pattern at `useAgent.ts:16-18`).
- UI state (`messages`, `streamingMessage`, `isStreaming`, `error`) is driven by the RPC event subscription, not by the `Agent` subscription directly.
- Public API of the hook unchanged — same return shape so `ChatDemo.tsx` et al. don't need to be touched.

### 1.5 Verify Phase 1

- Unit test under `packages/web-agent/src/web-agent/**/*.test.ts` using vitest for RPC round-trip: send a `prompt` command with a mocked `StreamFn` and assert event sequence.
- The existing Playwright chat test continues to pass: login → select model → send → reply contains "tuesday".
- No UI regressions — visually identical app.

## Phases 2–6 (roadmap sketch, will be planned in detail before each)

### Phase 2 — ZenFS mount + FSA picker

- add `@zenfs/core`, `@zenfs/dom`, `idb-keyval` deps
- `src/web-agent/fs/zenfs-provider.ts`: `mountVault(handle)` / `unmountVault()` wrapping `@zenfs/dom` `WebAccess`
- `src/hooks/useDirectoryHandle.ts` (app-side): pick folder, persist handle in IndexedDB, re-request permission on reload
- minimal UI: folder picker button + current vault badge in header
- **dev seam**: `useDevSeedBoot()` gated by `import.meta.env.DEV` reads `window.__zenfsSeed` populated by Playwright's `installVault(page, name)` helper, mounts an InMemory backend before React renders. This is the key lever that lets us run fs tests headlessly without the user-gesture picker. Source pattern: `bodhiapps/zenfs-browser/src/hooks/useDevSeedBoot.ts` + `e2e/helpers/install-vault.ts`.
- e2e: 1 test that seeds a 3-file vault, confirms the sidebar lists them.

### Phase 3 — filesystem tools

- port tool schemas from `packages/coding-agent/src/core/tools/{read,write,edit,ls,glob,grep}.ts` into `src/web-agent/core/tools/`, swap node ops for ZenFS ops via the existing `Operations` pattern (coding-agent already supports pluggable ops — we reuse the interface, not the package)
- bash/exec not ported (out of scope; will return "not available" if invoked)
- agent gains these tools automatically via `sendMessage` wiring
- e2e: 1 test — seed vault with `hello.txt` containing "world", prompt agent to read then write a new file, assert file contents via post-test read of the seeded InMemory FS

### Phase 4 — RPC hardening + Worker transport

- move `AgentSession` + tool execution into a Web Worker
- replace in-process transport with Worker + MessagePort transport (same interface)
- confirm ZenFS mount location: mount on main thread and expose via a proxy, OR mount in Worker directly — decide based on FSA handle transferability (structured-clone of `FileSystemDirectoryHandle` is supported; can be posted to worker)
- no new e2e; existing tests must continue to pass unchanged

### Phase 5 — extensions

- `/extensions` mount: separate ZenFS mount (IndexedDB backend) persisting downloaded extension bundles
- extension manifest JSON schema: name, version, entry (ESM), permissions, hooks
- loader: static-registered built-ins first; then support loading a user-provided manifest URL → download → persist to `/extensions/<name>/` → load entry as ESM blob URL in a sandboxed Worker
- hooks/interceptors mirror `coding-agent/core/extensions/runner.ts`: `before/after` tool call, turn start/end, custom tool registration
- security: each extension in its own Worker, message-passed; no direct FS access — they request via capability-gated RPC back to host
- e2e: 1 test — install a trivial "uppercase-echo" extension, prompt agent, verify extension hook fires and mutates tool output

### Phase 6 — extraction prep

- freeze the public API of `src/web-agent/`
- extract `peerDependencies` (`react` stays in the app, not in the agent package)
- add `package.json` stub + separate build pipeline so `src/web-agent/` can graduate to `packages/web-agent-core` or similar
