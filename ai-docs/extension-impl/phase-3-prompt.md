# M8 — Extensions — Phase 3 handoff prompt

> **STATUS: DEFERRED — no longer on the roadmap.**
>
> The product decision is that installed extensions are **fully trusted**.
> Once a user adds an extension to their vault, it executes as first-class
> code inside the agent Worker with the same capabilities as core. No
> iframe sandbox, no Worker-per-extension, no capability broker.

## Context

Phase 1 landed the browser-native extension runtime (Blob-URL factory
loader, `before_agent_start` / `tool_result` hooks, `registerTool` +
`registerCommand`, enable-map with broken-extension surfacing).

Phase 2a landed every context/lifecycle hook coding-agent exposes minus
compaction (`on('context')`, `on('tool_call')`, `on('turn_start')`,
`on('message_end')`, `on('session_loaded')` fired from `/reload` only) plus
the modal-only `pi.ui.*` channel (`notify`, `setStatus`, `select`,
`confirm`, `input`).

Phase 2b closed the remaining hook / registration / UI surface gap
against coding-agent. What is **not** landed: extension isolation
(everything still runs inline in the agent Worker), TypeScript / bundler
support, the `registerKeybinding` hook, and a name-collision UX for
overlapping skill / command / provider ids. Phase 3 picks those up.

Phase 2b landed:

- Widened `session_loaded.reason` to
  `'mount' | 'reload' | 'switch' | 'fork' | 'new' | 'navigate'`, fired
  from every `WorkerAgentHost` session-transition path.
- Added `before_compact` (reducer; returns `{ cutIndex?, preserveEntries? }`)
  and `after_compact` (observer) hooks, wired through
  `WorkerAgentHost.runCompaction`.
- Added `pi.registerProvider` / `pi.registerSkill` registration helpers
  backed by `ExtensionProviderController` (composite `LlmProvider`, churn
  emits `extension_providers_changed`) and `ExtensionSkillController`
  (in-memory SKILL.md bodies pushed into `CommandRegistry` with
  `source: 'extension-skill'`).
- Added `ctx.session` — a `ReadonlySessionForwarder` that re-reads the live
  `SessionManager` every call and throws `InvalidSessionError` on stale
  reads.
- Extended the `pi.ui.*` channel with `setTitle` (chat-header slot),
  `setWidget` (closed enum `progress | info | choice` in a transcript
  slot), `editor` (modal `EditorDialog`), and `setEditorText` (fire-and-forget
  buffer mutation).
- Added main-thread components (`ExtensionTitleSlot`, `ExtensionWidgetSlot`,
  `EditorDialog` branch in `ExtensionUIRenderer`) and extended
  `useExtensionUI` to manage their state.
- Added six new fixtures (`title-marker`, `progress-widget`, `note-editor`,
  `echo-provider`, `compaction-nudger`, `skill-nudge`) and a dedicated
  `e2e/extensions-ui-2b.spec.ts` covering DOM witnesses.
- Split the Phase 1 e2e vault: the original `extensions.spec.ts` now uses
  `e2e/data/sample-phase-1-extensions/` (minimal — 4 fixtures) so the one
  LLM-coupled step is not perturbed by Phase 2a/2b hook noise;
  `sample-with-extensions/` (15 fixtures) continues to back the UI specs.
- Published [`ai-docs/extension-guide/`](../extension-guide/README.md) —
  user-facing walkthrough per fixture so the test extensions double as
  discoverable capability demos.

Phase 3 is the **isolation / runtime expansion** phase. The web-agent
extension feature surface now matches coding-agent on hooks, registration,
and UI; what's left is the trust model, the bundler story, and the final
missing `pi-tui`-only features (keybindings, marketplace, richer editors).

## Goals for Phase 3

Pick the slice that gives the host user meaningful isolation without
forcing a ground-up rewrite. In priority order:

1. **Isolation model.** Pick one of:
   - **Iframe per extension** — null-origin sandbox, structured-clone RPC
     to the agent Worker.
   - **Worker per extension** — separate `Worker`, `MessageChannel` RPC to
     the agent Worker.
   - **Shared Worker + capability broker** — one additional Worker owning
     every extension; `pi` calls broker across.
   Write the decision in `phase-3-plan.md` with a rationale referencing
   `packages/web-agent/src/sandbox/` (the skill-sandbox precedent).
2. **TypeScript / bundler support.** Either ship a Worker-side transpile
   (e.g. `esbuild-wasm`) or formalise the "no TS" stance. If TS is added,
   keep source maps so extension errors still carry useful stacks.
3. **Keybinding hook.** Decide whether `pi.registerKeybinding` makes sense
   in a browser host. If yes, route through the main thread via a new RPC
   kind; if no, document the permanent divergence.
4. **Name-collision UX.** Today extension commands / skills / providers
   silently first-register-wins. Add a user-facing warning path (e.g.
   `ExtensionDescriptor.conflicts`) so users can diagnose overrides.
5. **Marketplace story (optional).** Not a hard requirement for Phase 3,
   but if we land a per-extension isolation model the discovery /
   distribution story becomes tractable.

Stretch, if the isolation pick is cheap:

- Editor UX: revisit inline editing if the isolation RPC can carry richer
  layout primitives without re-introducing function transfer.
- Visible dialog timeout countdown in `ExtensionUIRenderer`.

## Non-goals

- Introducing new hook kinds beyond what coding-agent exposes.
- Breaking Phase 2a / 2b RPC wire shapes; additive-only changes to
  `rpc-types.ts` unless a migration justifies a bump.
- Main-thread permissions / capability UI — Phase 3 is about the worker /
  isolation plumbing.

## Landed API shapes (Phase 2b)

The following shapes are stable and must not regress — Phase 3 changes
should be additive or migration-flagged.

### Hook surface

```ts
type ExtensionEvents = {
  before_agent_start: (e: BeforeAgentStartEvent) => BeforeAgentStartEventResult | void;
  tool_result: (e: ToolResultEvent) => ToolResultEventResult | void;
  context: (e: ContextEvent) => ContextEventResult | void;
  tool_call: (e: ToolCallEvent) => ToolCallEventResult | void;
  turn_start: (e: TurnStartEvent) => void;
  message_end: (e: MessageEndEvent) => void;
  session_loaded: (e: SessionLoadedEvent) => void;
  before_compact: (e: BeforeCompactEvent) => BeforeCompactEventResult | void;
  after_compact: (e: AfterCompactEvent) => void;
};

type SessionLoadedReason =
  | 'mount' | 'reload' | 'switch' | 'fork' | 'new' | 'navigate';
```

### Registration surface

```ts
pi.registerTool(def: ToolDefinition): void;
pi.registerCommand(name: string, opts: { handler, description?, argumentHint? }): void;
pi.registerProvider(provider: LlmProvider): void;
pi.registerSkill(skill: {
  name: string;
  description: string;
  body: string;
  disableModelInvocation?: boolean;
}): void;
```

### UI channel

```ts
pi.ui = {
  notify(message: string, type?: 'info' | 'warning' | 'error'): void;
  setStatus(text: string | null): void;
  setTitle(text: string | null): void;
  setWidget(slotKey: string, widget: ExtensionWidget | null): void;
  editor(
    title: string,
    prefill: string,
    opts?: { placeholder?: string; language?: string; signal?; timeout? }
  ): Promise<string | undefined>;
  setEditorText(text: string): void;
  select<T>(title, options: { label; value: T }[], opts?): Promise<T | undefined>;
  confirm(title, message, opts?): Promise<boolean>;
  input(title, placeholder?, opts?): Promise<string | undefined>;
};

type ExtensionWidget =
  | { kind: 'progress'; label?: string; value?: number; indeterminate?: boolean }
  | { kind: 'info'; title?: string; body?: string }
  | { kind: 'choice'; prompt: string; options: { id: string; label: string }[] };
```

### Context surface

```ts
type ExtensionContext = {
  readonly cwd: string;
  readonly ui: ExtensionUIContext;
  readonly hasUI: true;
  readonly session: ReadonlySessionManager; // throws InvalidSessionError on stale reads
  isIdle(): boolean;
  abort(reason?: string): void;
};
```

### RPC additions

- Event `extension_providers_changed` (`{ providers: ExtensionProviderDescriptor[] }`).
- `extension_ui_request.kind` widened to include `setTitle`, `setWidget`, `editor`, `setEditorText`.

## Hard constraints carried from Phase 2a / 2b

- **Structured-clone safety.** No functions / class instances / DOM refs over RPC. Every payload must survive `postMessage`.
- **Per-extension error isolation.** One broken extension must not take down the agent. Runner wraps every handler in `try/catch` + fans errors through `reportError`.
- **FIFO modal queue.** Every modal verb (`select`, `confirm`, `input`, `editor`) shares the same queue keyed by `requestId`. Cancel-on-session-reset resolves with the channel's cancel value.
- **Lazy hook install.** `setBeforeToolCall` / `setTransformContext` / `setAfterToolCall` install only when a runner handler appears. Do not pre-install.
- **Spec update in the same PR.** Any change to `core/extensions/**`, `worker/extension-*.ts`, `rpc/*.ts` (extension paths), or `src/components/extensions/**` updates `ai-docs/specs/worker-agent/extensions.md` in the same change.
- **No LLM-text assertions in e2e.** DOM / RPC / files-on-disk witnesses only.
- **No `packages/coding-agent` imports in `packages/web-agent`.**

## Open questions Phase 3 must pick a side on

- **Isolation target.** Iframe? Worker-per-extension? Shared Worker? The
  answer constrains the RPC fan-out and the bundler story.
- **TypeScript story.** Ship a Worker-side transpiler or keep JS-only?
- **Keybinding hook.** Browser-compatible or permanently deferred?
- **Name-collision surfacing.** Surface conflicts on `ExtensionDescriptor`?
  Or keep silent first-wins?

  UX, or is the modal dialog the final answer?
